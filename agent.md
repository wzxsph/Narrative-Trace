# 本项目 Agent 操作准则

本文档定义 agent 在 `/home/samsong/Desktop/game_writer` 项目中的默认工作规则。所有后续构建、修改、整理、调研和实现工作，都应优先遵守本文件。

## 1. 上下文读取规则

### 1.1 禁止读取给人类看的文档

不要主动读取以下目录及其子目录内容：

```text
/home/samsong/Desktop/game_writer/doc/给人类看的文档
/home/samsong/Desktop/game_writer/给人类看的文档
```

原因：

- 这些文档主要面向人类阅读，不是当前构建阶段的权威执行准则。
- 内容通常较长，读取会浪费 token。
- 若需要确认项目方向，应优先查看 `doc/prd`。

允许的例外：

- 用户明确指定读取其中某个文件。
- 用户要求整理、归档、迁移该目录中的文件。
- 当前任务无法在不读取该目录的情况下完成，并且已先尝试从 `doc/prd` 获取依据。

### 1.2 reference 目录定位

`reference` 是前期调研资料目录。

```text
/home/samsong/Desktop/game_writer/reference
```

使用原则：

- 可以作为背景资料、竞品资料、工具链参考、论文和开源项目参考。
- 不应把 `reference` 中的内容直接视为当前产品准则。
- 当 `reference` 与 `doc/prd` 冲突时，以 `doc/prd` 为准。
- 读取 `reference` 时要有明确目标，避免全量扫描大型源码、论文或导出文档。

### 1.3 PRD 是当前唯一产品准则

本项目当前阶段的唯一产品准则位于：

```text
/home/samsong/Desktop/game_writer/doc/prd
```

工作原则：

- 构建项目、设计功能、解释需求、拆分任务时，应优先向 `doc/prd` 对齐。
- 若用户需求与 PRD 存在冲突，应先指出冲突，再按用户最新指令执行或请求确认。
- 不要凭记忆或旧讨论覆盖 PRD 当前内容。
- 可以读取 PRD 来确认当前阶段范围、UI/UX、交互规则、验收标准和非目标。

## 2. PRD 修改规则

### 2.1 默认不要修改 PRD

构建项目时，避免更改 `doc/prd` 下的 PRD 文件。

PRD 是当前阶段的产品基准，不应在实现过程中被顺手改写、重排、润色或自动格式化。

### 2.2 允许修改 PRD 的情况

只有在以下情况下可以修改 PRD：

- 用户明确要求更新、重写、补充或归档 PRD。
- 用户要求把新的产品决策沉淀到 PRD。
- 当前任务本身就是 PRD 编辑任务。

修改 PRD 时：

- 保留旧版本，除非用户明确要求覆盖。
- 说明改动范围。
- 不要把实现细节误写成产品准则。

## 3. 构建与实现规则

### 3.1 先对齐 PRD，再动手实现

实现任何功能前，应先确认它属于 PRD 当前阶段范围。

重点对齐：

- 游戏框架。
- 用户 UI/UX。
- observe / choice 规则。
- 隐藏内容与嵌套展开。
- 解锁 choice 的条件。
- 章节路径图和复盘体验。
- 当前阶段明确不做的内容。

### 3.2 避免范围膨胀

不要在没有用户明确要求时主动加入：

- AI 自动生成完整游戏。
- 作者工具后台。
- 商业化系统。
- 社区、账号、排行榜、分享增长功能。
- 大规模工程化重构。
- 与当前 V0 无关的复杂技术栈。

如确实认为某项扩展必要，应先说明理由和风险。

### 3.3 以当前阶段可验证体验为优先

本项目当前更重视：

- 用户是否理解文字可以展开。
- 用户是否能通过 observe 解锁 choice。
- 用户是否感到选择有后果。
- 用户是否能在路径图中复盘未探索路径。

不要为了技术完整性牺牲第一版体验闭环。

## 4. 文件操作规则

### 4.1 新增文件位置

新增项目文档时，优先放入合适的 `doc` 子目录。

新增实现文件时，应根据实际技术栈建立清晰目录，不要混放到 `reference` 或 `doc/给人类看的文档`。

### 4.2 不要污染 reference

`reference` 是调研资料归档区，不应放入当前项目实现代码、临时输出或新的产品准则。

### 4.3 谨慎移动文件

移动、重命名、归档文件前，先确认目标位置和影响范围。

不要移动 PRD，除非用户明确要求。

## 5. 迭代日志规则

### 5.1 每次迭代必须记录 log

每次进行有意义的项目迭代后，都要在以下目录新增或更新一条 Markdown 日志：

```text
/home/samsong/Desktop/game_writer/log
```

日志命名建议：

```text
YYYY-MM-DD-HHMMSS-mmm-short-topic.md
```

其中 `HHMMSS` 使用 24 小时制本地时间，`mmm` 为毫秒。默认使用工作环境本地时区生成，例如：

```bash
date '+%Y-%m-%d-%H%M%S-%3N'
```

这样同一分钟甚至同一秒内多次迭代也能保持清晰顺序，避免日志文件名冲突。历史日志如果只有秒级时间，可用 `000` 补齐毫秒位，表示原始记录没有更细时间。

日志至少包含：

- 本次迭代的开始时间和结束时间，格式建议为 `YYYY-MM-DD HH:MM:SS.mmm +0800`。
- 本次迭代做了什么。
- 新增、修改、移动或删除了哪些文件。
- 做过哪些验证。
- 是否有遗留问题或下一步建议。

如果一次迭代内发生多个关键步骤，可在日志正文中用 `HH:MM:SS.mmm` 记录关键节点，便于复盘生成、验证、修复之间的顺序。

### 5.2 log 不替代最终回复

写入 log 后，最终回复仍需简要说明本次交付内容和关键文件路径。

### 5.3 log 不应泄露敏感信息

不要在 log 中写入：

- API key。
- `.env` 内容。
- 任何不应进入仓库的本地私密配置。

## 6. 回答与交付规则

### 6.1 交付时说明依据

完成任务后，应简要说明：

- 做了什么。
- 依据了哪个准则或 PRD。
- 哪些文件被新增、移动或修改。

### 6.2 发现冲突时先指出

如果用户要求、PRD、reference 或现有文件之间出现冲突：

1. 明确指出冲突。
2. 说明推荐处理方式。
3. 若冲突影响较大，再请求用户确认。

### 6.3 不要过度解释

除非用户要求，最终回复保持简洁，以文件路径和关键结果为主。

doc/prd 是唯一产品准则。
构建项目时经常对齐 doc/prd。

每次有意义迭代后记录 log。
