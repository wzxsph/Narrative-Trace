版本：V1.0
日期：2026-07-20
文档性质：框架层正式定义文档
文档目标：定义一个契约固定、玩法可插拔、媒介可扩展、人类与 agent 同权创作的互动叙事游戏框架
本文档不定义：具体实现计划、代码重构方案、商业化系统、UGC 平台运营、聚合分发入口

冻结记录：2026-07-20 完成谈判循环纸面扩展测试与 text→image Surface 等价测试，六项契约无需修改，V1.0 正式冻结。冻结前版本归档于 `doc/framework/old version/2026-07-20-v1-0-draft/`。

## 1. 一句话定义

> 一个以"隐藏信息被玩家亲手揭开"为核心资产的互动叙事游戏框架：契约层保证任何作品可验证、可存档、可复盘、可追溯；玩法包保证交互循环经过市场检验；无论人类还是 agent 填充内容，都必须穿过同一套验证闭环才能到达玩家。

更短一点：

> 框架不保证故事好看，框架保证：能点开的东西是公平的，做过的事会被记住，没验证过的东西不许说自己验证过。

---

## 2. 市场认可：框架的第一性目标

### 2.1 定义

"被市场认可"在本框架中只有一个含义：

> 基于框架产出的具体作品，被玩家用行为投票认可。

框架本身不直接面对市场。框架的每一个机制都必须能回答同一个问题：

```text
它如何提高作品被玩家认可的概率？
```

回答不上来的机制不进入契约层。

### 2.2 指标口径

V1 阶段以内部 playtest 批次（5 到 8 人）为测量单位，口径沿用并扩展 PRD V0 第 14 节：

| 指标 | 口径 | 数据来源 |
|---|---|---|
| 完成率 | 开始游玩的用户中完成首周目的比例 | playtest 记录 |
| 主动发现率 | 关键 observe 被玩家自主发现的比例 | 游玩痕迹 |
| 因果理解率 | 能说出至少一个 "observe 解锁 choice" 实例的用户比例 | playtest 访谈 |
| 后果感知率 | 能说出至少一个选择的后续影响的用户比例 | playtest 访谈 |
| 重玩意愿 | 查看路径图后表达重玩意愿的用户比例 | playtest 访谈 |
| 卡死率 | 因关键隐藏点不可发现而卡死的用户数（必须为 0） | playtest 记录 |
| 付费/传播意愿 | 愿意付费或主动推荐的用户比例（V1 仅问卷口径） | playtest 访谈 |

线上真实数据（留存、付费、传播）是这些口径的后续演进，V1 只定义口径，不实现采集（见第 11 节范围切割）。

### 2.3 认可如何回流到框架

```text
玩家数据认可作品 -> 作品数据为其玩法包积累证据 -> 玩法包获得/保持 Verified 等级
玩家数据否定作品 -> 先归因（内容问题 or 玩法问题）-> 玩法问题降级玩法包并修包
```

框架的长期资产不是某个作品，而是一组"已被玩家数据验证过的玩法包"。

## 3. 设计原则

1. **作品先于框架。** 框架机制的取舍标准是作品的市场表现，不是工程完备性。
2. **契约固定，玩法可插拔。** 六项契约（第 5 节）是所有作品的共同底座；交互循环、节奏、动词由玩法包定义，可替换。
3. **观察语义与媒介无关。** "发现锚点、展开内容、写入状态、解锁行动"这套语义在文字、图片、HTML 界面上必须同构（第 6 节）。
4. **验证优先于生成。** 门禁先于内容存在；人类与 agent 产出的内容穿过完全相同的门禁，不存在 agent 特权通道，也不存在人类特权通道。
5. **诚实分级。** 未经真实玩家数据验证的玩法不得标记为 Verified；Experimental 作品必须对玩家可见地标注。框架宁可承认"未验证"，不制造质量幻觉。

## 4. 四层架构

### 4.1 总览

```text
┌────────────────────────────────────────────────────┐
│ L4 运行时与分发层 Runtime & Distribution            │
│ 各 surface 渲染器 · 存档执行 · 复盘 UI · 静态部署    │
├────────────────────────────────────────────────────┤
│ L3 内容包层 Content Packs                           │
│ 具体作品：剧本数据 + 素材 + 状态实例 + 来源追溯       │
│ 生产者：人类作者 / 人类引导下的 agent                │
├────────────────────────────────────────────────────┤
│ L2 玩法包层 Loop Packages                           │
│ 玩法定义 + schema 扩展 + 验收规则集 + 体验指标        │
│ 分级：Verified / Experimental                       │
├────────────────────────────────────────────────────┤
│ L1 契约层 Kernel Contracts（不变量）                 │
│ 状态 · Surface · 行动 · 进程 · 存档 · 验收与追溯      │
└────────────────────────────────────────────────────┘
```

### 4.2 各层职责

**L1 契约层**：与题材、媒介、玩法、供应商都无关的六项契约。它是框架的不变量：修改 L1 等于修改框架版本，需要走框架级决策，任何玩法包或作品不得私自绕过。

**L2 玩法包层**：定义"玩家到底在反复做什么"。一个玩法包 = 一种交互循环 + 它对契约层的 schema 扩展 + 它自带的验收规则 + 它的体验指标。玩法包是可插拔组件：调查循环是第一个，经营、谈判、解谜等是未来的候选。

**L3 内容包层**：具体作品。一个内容包必须声明它基于哪个玩法包（且只能一个），并同时满足 L1 契约和该玩法包的扩展 schema。

**L4 运行时与分发层**：把合法内容包变成玩家可玩的东西。按 surface 类型提供渲染器，执行存档契约，渲染复盘，并支持独立静态部署。

### 4.3 依赖规则

```text
L3 依赖 L2 依赖 L1；L4 依赖 L1（渲染约定另见玩法包声明）。
只允许上层依赖下层，禁止反向依赖。
状态读写只能通过 L1 状态契约的效果原语进行，任何层不得私设状态通道。
一个内容包声明且只声明一个玩法包。
```

### 4.4 角色与层的关系

| 角色 | 主要活动层 | 权限边界 |
|---|---|---|
| 玩家 | L4 | 只接触运行时，看不到契约与门禁 |
| 人类作者 | L3 | 生产内容包，可用任何等级玩法包 |
| agent | L3 | 默认只能面向 Verified 玩法包生产（见 7.5） |
| 玩法设计者 | L2 | 定义新玩法包，必须交齐四件套（见 7.1） |
| 框架维护者 | L1 + L2 验收规则 | 契约与门禁的唯一修改者 |

## 5. 契约层：六项契约

契约层的交付物形态是：一组 JSON Schema + 一组可机读的校验规则 + 本节的语义说明。以下定义语义与必填能力，字段级 schema 属于实现阶段。

### 5.1 状态契约 State

世界如何记住玩家。

- 状态必须在内容包的**状态注册表**中先声明后使用（现有工程的 `state_registry.json` 是其雏形）。
- 状态分四个语义族，沿用 PRD V0 第 9 节：`clue`（知道什么）、`stance`（倾向什么）、`relationship`（别人怎么看你）、`pressure`（外部压力）。
- 写状态只能通过**效果原语**（`set` / `add`），读状态只能通过**条件原语**（`equals` / 阈值比较）。
- 状态对玩家默认不可见；对玩家的呈现只能走**回声**（`state_echoes`：满足条件时插入的叙事文本），不得显示变量名和数值。
- 未注册状态、未被任何条件读取的死状态、被读取但从未被写入的空状态，都是验收错误。

### 5.2 Surface 契约 Surface

玩家观察什么。这是 V1 相对现有工程最重要的抽象升级，详见第 6 节。核心定义：

- 一个场景由若干 **surface** 组成；每个 surface 有类型：`text` / `image` / `html`。
- surface 上分布 **anchor**（观察锚点）；anchor 的定位方式随媒介不同，但语义完全一致。
- anchor 被玩家发现并展开后产生 **fragment**（展开内容），fragment 内可以继续嵌套 anchor。
- anchor 必须声明可发现性等级：`obvious` / `subtle` / `hidden_optional`。
- **公平性公理（契约级）**：解锁 `chapter` / `global` / `ending` 级后果行动的 anchor，可发现性不得为 `hidden_optional`。玩家可以漏掉线索，但不能因为界面不公平而漏掉。

### 5.3 行动契约 Action

玩家如何改变世界。

- 行动的必填语义：`id`、短标签、后果描述、`requirements`（条件原语）、`effects`（效果原语）、去向、`irreversible` 标记、后果等级（`local` / `chapter` / `global` / `ending`）。
- requirements 未满足的行动对玩家不可见或明确锁定，由玩法包决定呈现方式，但**不得出现"可点击却无意义"的行动**。
- 不可逆行动必须有叙事化提示（"这会留下记录"），禁止系统化警告（"将修改变量 X"）。
- observe 与 action 的边界是契约级的：观察改变玩家知道什么（可逆），行动改变世界状态（通常不可逆）。玩法包可以重命名这两个动词，不得合并它们。

### 5.4 进程契约 Progression

作品的骨架与终点。

- 内容包必须声明：入口场景、场景图（节点与去向）、至少一个结局。
- 所有结局必须可达（机器可证明），所有场景必须从入口可达。
- 每个结局必须携带**画像标签**：回答"保护了谁、伤害了谁、相信了什么"，供结局复盘渲染。
- 运行时必须持续记录**游玩痕迹**（visited 场景、已发现 anchor、已解锁/已选择行动、状态快照），痕迹数据结构属于契约；如何把痕迹渲染成路径图/复盘属于玩法包与运行时。

### 5.5 存档契约 Save

进度如何幸存。沿用现有工程 V0.21–V0.24 已验证的合同：

- 存档必须带版本号；恢复流程固定为：**先迁移，再校验，失败则 fallback 到新局并给出用户可见的恢复提示**。
- 坏 JSON、未知未来版本、引用不存在场景的存档，都必须落入 fallback，不得卡死。
- 存档按 内容包 id + 版本 隔离命名空间。
- 每个运行时实现必须通过存档合同 fixture 回放（现有 `examples/fixtures/save_contract/` 是参考实现）。

### 5.6 验收与追溯契约 Verification & Provenance

什么东西可以声称自己是合法的。

- 契约层定义门禁的**接口**：每道门禁是一个可机读运行、输出 error/warning 的检查器（门禁链见第 10 节）。
- 内容包必须携带**生成追溯**：由 agent 参与生成的内容包，`provenance` 必填 `prompt_set`、`provider`、`model`、`schema` 版本、trace 版本（现有 `generation_trace.jsonl` 是参考实现）；纯人工内容包标注 `human_authored`。
- 模型原始输出入库必须走脱敏归档合同（checksum、敏感信息扫描），沿用现有 V0.17/V0.18 机制。
- 任何门禁结果可复现：同一内容包 + 同一门禁版本 = 同一结论。

## 6. Surface 抽象

### 6.1 三类 surface 与锚点定位

| surface 类型 | 内容形态 | anchor 定位方式 | 典型来源 |
|---|---|---|---|
| `text` | 叙事文本块 | 文本范围（span） | 人类写作 / LLM 生成 |
| `image` | 图片 | 归一化区域（矩形/多边形，0–1 坐标） | 美术素材 / 文生图模型 |
| `html` | 受限 HTML 界面（如一部"手机屏幕"、一张网页） | 元素选择器（受限白名单标签内） | 人类设计 / LLM 生成界面 |

### 6.2 媒介无关的观察语义

三类 surface 共享完全相同的语义链：

```text
玩家察觉异常 -> 点击 anchor -> 展开 fragment -> （可选）写入状态 -> （可选）解锁行动 -> fragment 内可继续嵌套
```

对内容创作者（人或 agent）而言，"在一段文字里埋'02:13'"和"在一张桌面照片里埋'半开的抽屉'"是同一个创作动作，产出同一种数据结构，走同一套验收。

fragment 本身也是 surface 的容器：文字锚点可以展开出一张图片证据，图片锚点可以展开出一段文字日志。**素材兼容性由此获得：媒介在 surface 层自由混合，状态与行动层完全感知不到媒介差异。**

### 6.3 公平性规则的媒介无关化

5.2 的公平性公理在三类媒介上的落点：

- `text`：关键 anchor 必须有文本异常或视觉暗示（下划线、留白、语气异常）。
- `image`：关键 anchor 区域必须有可察觉的视觉暗示（构图引导、光线、违和物件），且区域尺寸不得小于可触控下限。
- `html`：关键 anchor 元素必须在无 hover 的触屏环境下可被察觉，不得依赖 hover 态或纯色差。

"关键线索不能藏成像素猎巫"是契约，具体检查手段随媒介由验收规则实现。

### 6.4 替换与降级

同一内容包中，一个 surface 从 `text` 替换为 `image`（例如把文字描述的桌面换成 AI 生成的桌面图片），只改动 surface 定义与 anchor 定位，**状态、行动、进程、存档契约零改动**。这是 Surface 抽象是否成立的硬标准（见第 13 节成功标准 4）。

运行时若不支持某 surface 类型，必须声明降级策略（如 image 降级为替代文本），不得静默丢弃 anchor——丢弃 anchor 等于破坏公平性。

### 6.5 V1 边界

- V1 规范**定义全部三类 surface 的契约**（数据结构、定位方式、公平性落点、降级策略）。
- V1 只要求 `text` 有参考实现（现有 Narrative Trace 引擎升级而来）。
- `image` / `html` 的参考实现是后续里程碑，不阻塞框架定稿；但契约设计时必须用 6.4 的替换推演做纸面验证。

## 7. 玩法包

### 7.1 四件套

一个玩法包必须交齐四件套，缺一不是玩法包，只是想法：

```yaml
LoopPackage:
  loop_definition:      # 1. 玩法定义
    core_verbs: []      #    玩家反复执行的动词（如 observe / choose）
    loop_diagram: ...   #    循环结构与节奏预算（每节点多长时间在做什么）
    boundary: ...       #    明确不做什么
  schema_extension:     # 2. 内容 schema 扩展
    ...                 #    在六项契约之上追加的字段与约束（不得削弱契约）
  acceptance_rules:     # 3. 验收规则集（全部机器可运行）
    structure: []       #    结构校验器
    content_qa: []      #    内容 QA 规则（公平性、节奏、死内容检测）
    simulation: []      #    模拟通关脚本
    e2e: []             #    浏览器级 E2E 场景
  experience_metrics:   # 4. 体验指标
    metrics: []         #    指标定义与达标阈值
    playtest_template:  #    playtest 记录模板与批次汇总口径
```

### 7.2 分级：Verified / Experimental

| 等级 | 含义 | agent 可否默认填充 | 作品标注 |
|---|---|---|---|
| **Verified** | 四件套齐全 + 自动门禁全绿 + 至少一个作品的真人 playtest 达到自身指标阈值 | 可以 | 无需特殊标注 |
| **Experimental** | 四件套齐全 + 自动门禁全绿，但缺真人数据 | 仅在人类显式 opt-in 时 | 必须对玩家可见地标注"实验性玩法" |

四件套不齐的东西不进入分级体系，不允许生产对外作品。

### 7.3 晋升与降级

```text
晋升：Experimental -> Verified
条件：基于该玩法包的至少一个作品，完成一轮真实 playtest 批次（5-8 人），
     全部达标其体验指标，且批次原始记录已归档可查。

降级：Verified -> Experimental
条件：后续作品的玩家数据系统性未达标，且归因为玩法问题而非单一作品的内容问题。
降级后修包重走晋升。
```

刷数据防线：晋升依据的 playtest 批次必须使用玩法包自带模板逐人记录并归档，汇总结论必须能从原始记录复算（现有 `summarize_playtest_batch.py` 合同）。

### 7.4 第一个玩法包：调查循环（Investigation Loop）

从 Narrative Trace / PRD V0 抽象而来：

```yaml
InvestigationLoop:
  core_verbs: [observe, choose]
  loop: 观察 -> 解读 -> 行动 -> 状态 -> 回声 -> 复盘
  节奏预算: 30s 理解处境 / 60-120s 探索 / 15-30s 决策 / 10s 后果反馈
  schema_extension:
    - observe 嵌套深度上限 3
    - 每主场景至少一个 obvious anchor
    - observe 解锁 choice 的 unlock contract（声称解锁则 effects 必须能满足 requirements）
    - 章节复盘与路径图数据要求
  acceptance_rules: 现有 validate_game / content_qa_report / smoke_playthrough / browser 全家桶的抽象
  experience_metrics: PRD V0 第 14.2 节量化指标
```

**等级判定（诚实声明）**：调查循环被指定为首个 Verified 玩法包，但其 Verified 状态**带 playtest 债务**——PRD 17.2 明确 5–8 人内部测试尚未完成。债务落账规则：

- 在首轮真实 playtest 批次归档前，该包状态记为 `Verified(债务)`，允许 agent 默认填充（否则框架冷启动死锁）。
- 首轮 playtest 完成且达标 → 债务清除，成为无条件 Verified。
- 首轮 playtest 未达标 → 立即降级 Experimental，框架的第一优先事项变为修包。

这是整个框架 V1 最重要的一次真实检验。

### 7.5 agent 权限边界

- agent 默认只能面向 Verified（含带债务的 Verified）玩法包生产内容包。
- 人类显式 opt-in 后，agent 可为 Experimental 包生产，产物强制携带实验标注。
- agent 不得修改任何玩法包的验收规则与指标阈值——那是框架迭代，属于人类决策。
- agent 不得绕过门禁发布；门禁全绿也不等于发布，V1 阶段发布决定权在人类。

## 8. 内容包与分发

### 8.1 内容包结构

```text
content-pack/
  pack.json            # manifest（见 8.2）
  game.json            # 进程 + 状态 + 行动 + surface/anchor 数据
  assets/              # image / html surface 素材（text-only 作品可为空）
  provenance/          # 生成追溯：generation_trace.jsonl、模型输出样本引用
```

### 8.2 manifest 最小字段

```yaml
pack.json:
  pack_id: string            # 全局唯一
  title: string
  version: string
  loop_package:
    id: string               # 声明且只声明一个
    tier: verified | experimental
  surfaces_used: [text]      # 声明用到的 surface 类型，运行时据此判断兼容性
  runtime_compat: string     # 兼容的契约/运行时版本
  authorship: human | agent_assisted | agent_generated
  experimental_notice: bool  # experimental 玩法包作品强制为 true
```

### 8.3 独立分发

- 一个作品 = 内容包 + 引用的运行时，可部署到任意静态托管（GitHub Pages、Cloudflare Worker、自有服务器），无中心平台依赖。现有双部署即此模式的实例。
- "平台化"在 V1 的全部含义是：**契约标准化，使任何合法内容包可以被任何合规运行时加载**。
- 聚合入口（统一发现页、跨作品存档、玩家画像）是后续演进方向，V1 明确不做，仅要求 manifest 字段为未来聚合预留足够元数据。

## 9. Agent 创作管线契约

### 9.1 管线阶段

agent 是人类引导下的创作流水线，不是全自动作者。管线阶段与中间产物（artifact）都是契约的一部分，沿用现有工程已验证的结构：

```text
brief（人类拥有）
  -> state_schema_design   状态设计       + 校验器
  -> scene_blueprint       场景蓝图       + 校验器
  -> scene_artifacts       场景内容       + 校验器 + 蓝图对齐校验
  -> game.json + pack.json 内容包组装
  -> 门禁链（第 10 节）
```

每个阶段产物独立落盘、独立校验、失败可局部重跑，禁止"一口气生成整包"的黑箱模式——黑箱模式无法定位失败，也无法让人类在中途引导。

### 9.2 人类引导点

管线固定四个人类引导点：

1. **brief 确认**：题材、基调、规模、约束由人类拥有。
2. **蓝图审查**：scene_blueprint 生成后、批量生产场景内容前，人类可改方向（这是成本最低的干预点）。
3. **发布判断**：门禁全绿后是否发布，人类决定。
4. **playtest 解读**：指标达标与否的归因（内容问题还是玩法问题），人类决定。

其余环节 agent 自主执行，包括自跑 G1–G5 门禁与局部修复（repair 只修结构硬伤，不重写剧情，沿用现有 repair 边界）。

### 9.3 追溯

- 每次生成写入 trace：`prompt_set` / `provider` / `model` / `schema` / trace 版本。
- 真实模型输出入样本库必须脱敏 + checksum（现有归档合同）。
- 失败样本与 prompt 版本绑定，形成回归库：同一失败不许第二次静默出现。

## 10. 验证闭环

### 10.1 门禁链

每个内容包发布前必须穿过六道门禁，前五道机器自动、agent 可自跑，第六道有人类参与：

| 门禁 | 检查什么 | 属于 | 现有参考实现 |
|---|---|---|---|
| G1 Schema | 字段形状、类型、枚举、必填 | L1 契约 | `validate_json_schema.py` |
| G2 结构 | 可达性、引用完整性、状态注册、公平性公理 | L1 契约 | `validate_game.py` |
| G3 内容 QA | 玩法包的内容规则（unlock contract、节奏、死内容） | L2 玩法包 | `content_qa_report.py` |
| G4 模拟通关 | 全结局可达的确定性模拟 | L2 玩法包 | `smoke_playthrough.py` |
| G5 浏览器 E2E | 真实运行时行为：多结局、遗漏路径、存档合同、a11y、移动视口 | L4 运行时 | `browser_*.py` 全家桶 |
| G6 真人 playtest | 体验指标达标 | L2 指标 + 人 | playtest 模板与批次汇总 |

发布规则：

```text
G1-G5 全绿 = 允许发布（作品可以到达玩家）
G6 达标   = 作品记入其玩法包的验证证据（影响玩法包分级）
G6 不达标 = 作品可以继续存在，但玩法包晋升不得引用它；系统性不达标触发降级评估
```

### 10.2 数据回流

作品端游玩痕迹（完成率、anchor 命中率、choice 分布、复盘打开率、重玩率）按 2.2 口径汇入玩法包证据库。V1 阶段数据来源仅限 playtest 批次的人工记录 + 本地痕迹导出，**不建线上采集后端**；隐私边界（收集什么、征得什么同意）在实现采集前必须先行定义。

## 11. V1 范围切割

### 11.1 V1 做

- 六项契约的完整定义（本文档 + 后续字段级 schema）。
- 三类 surface 的契约定义；text surface 参考实现。
- 玩法包规范（四件套 + 分级 + 晋升/降级规则）。
- 调查循环包：从现有工程抽象为第一个玩法包，并完成 playtest 债务落账。
- agent 管线契约与四个人类引导点。
- G1–G6 门禁链定义（G1–G5 已有参考实现基础）。
- 内容包 manifest 与独立分发约定。

### 11.2 V1 不做

- 统一玩家门户 / 内容商店 / 聚合发现页。
- 账号、云存档、跨作品玩家画像。
- 商业化与分成系统。
- 面向外部创作者的 UGC 后台与可视化编辑器。
- image / html surface 的参考实现（仅契约 + 纸面推演）。
- 全自动无人创作（去掉四个人类引导点）。
- 线上数据采集后端。
- 第二个玩法包的实现（仅用于纸面推演检验契约）。

## 13. 框架成功标准

V1 框架定义成立，当且仅当以下四条全部通过：

1. **纸面扩展测试**：用本规范推演第二个玩法包（如谈判循环或经营循环），六项契约的任何必填语义无需修改。若需要修改，契约设计失败，回炉。
2. **agent 生产测试**：agent 仅凭（本规范 + 调查循环包 + 一份新 brief）产出的内容包能通过 G1–G5，且人类引导点不超过四个。
3. **市场测试**：首个按框架口径落账的作品完成 5–8 人 playtest，达到调查循环包指标阈值（即 7.4 债务清偿）。
4. **素材兼容测试**：把某作品的一个 text surface 替换为 image surface，状态/行动/进程/存档契约零改动。V1 阶段纸面推演通过，image 参考实现落地后实测通过。

冻结时状态：成功标准 1 的谈判循环纸面推演与成功标准 4 的契约级 text→image 替换测试已通过；成功标准 2 等待生成管线 V1 验收，成功标准 3 等待 5–8 人真人 playtest。

## 14. 风险与开放问题

| # | 风险 | 说明 | 对策 |
|---|---|---|---|
| 1 | 单实例抽象 | 六项契约目前只由一个玩法包归纳而来，抽象可信度有限 | 成功标准 1 的纸面推演是硬门槛；第二个玩法包实现前不冻结契约字段级 schema |
| 2 | image/html 公平性无先例 | "图片里的关键线索必须可发现"如何机器检查，业界无现成答案 | V1 只定契约；实现前先做小规模人工标注实验 |
| 3 | 分级被刷 | 熟人 playtest 可能刷过晋升门槛 | 批次原始记录归档可复算；后续引入线上数据交叉验证 |
| 4 | agent 作品同质化 | Verified 包收敛可能让 agent 作品千篇一律 | 靠题材/surface 组合多样性缓解；Experimental 通道保持玩法创新供给 |
| 5 | 数据与隐私 | 游玩痕迹回流涉及玩家行为数据 | V1 不建采集后端；采集前先定义隐私边界 |
| 6 | 命名未定 | "框架/引擎/平台"对外叙事与正式名称未定 | 暂用 FRAMEWORK V1 代称，命名是发布前决策，不阻塞设计 |

开放问题（不阻塞 V1 定稿，记录待议）：

- html surface 的安全边界：受限标签白名单的具体范围，脚本是否一律禁止。
- 多玩法包混排：一个作品能否分章使用不同玩法包（V1 答案：不能，一包一作品；未来重估）。
- 玩法包版本化：包升级后旧作品的兼容策略。
